# BNO055 IMU — Reference for Talos

---

## Conceptual Overview: What is a 9-DOF IMU?

An IMU (Inertial Measurement Unit) measures how a body is moving through space. "9-DOF" means nine degrees of freedom — three axes each of acceleration, rotation, and magnetic field.

The BNO055 contains three physical sensors:

| Physical sensor | Measures | Raw output |
|---|---|---|
| Accelerometer | Forces on the chip (gravity + linear motion) | m/s² on X, Y, Z |
| Gyroscope | Angular rotation rate | deg/s on X, Y, Z |
| Magnetometer | Earth's magnetic field | µT on X, Y, Z |

These raw sensor values are noisy and each has different failure modes:
- **Gyroscope** is fast and accurate in the short term but has *bias drift* — a tiny constant error (~0.01°/s) that accumulates. If you integrate gyro rate to get angle, you accumulate degrees of error over minutes.
- **Accelerometer** gives an absolute tilt reference via gravity (gravity always points down) but picks up every vibration and linear jolt as noise. Useless for fast-moving angle estimation alone.
- **Magnetometer** gives absolute heading (like a compass) but is disturbed by nearby motors and steel.

---

## Sensor Fusion: What the BNO055 Does Internally

The BNO055 contains a dedicated ARM Cortex-M0 processor running Bosch's proprietary fusion library. This firmware reads all three sensors and combines them in a Kalman-like complementary filter:

```
Gyroscope    →  fast, accurate rate          ──┐
Accelerometer →  absolute tilt (slow/noisy)  ──┤──► FUSION ──► Euler angles (stable, no drift)
Magnetometer →  absolute heading             ──┘
```

**The result:** fused orientation outputs that are accurate at all timescales — gyroscope handles fast motion, accelerometer corrects long-term gyro drift.

This happens entirely on the BNO055 chip. Your microcontroller just reads the result over I2C.

## Operation Modes

The BNO055 supports several fusion modes. You select via the OPR_MODE register (0x3D).

| Mode | Sensors used | Notes |
|---|---|---|
| CONFIG (0x00) | None | Required before changing settings |
| IMU (0x08) | Accel + Gyro | Relative orientation. No mag calibration needed. Yaw drifts slowly; pitch/roll stable. **← Talos uses this** |
| COMPASS (0x09) | Accel + Mag | Heading only |
| M4G (0x0A) | Accel + Mag (as gyro substitute) | Slow gyro replacement |
| NDOF_FMC_OFF (0x0B) | All three, mag uncalibrated | |
| NDOF (0x0C) | All three, full fusion | Absolute orientation in all axes. Requires magnetometer calibration. Best for absolute heading. |

**IMU mode** is correct for Talos Phase 2–5. Pitch and roll are fully accurate (gravity is the reference). Yaw drifts — but we don't need absolute yaw for balance. NDOF would add magnetometer-based yaw accuracy, at the cost of requiring magnetometer calibration whenever the robot is near motors or ferrous material. Not worth it for balance.

## All Available Outputs and How to Get Them

**Everything is in registers, readable over the same two I2C wires already connected.** No additional pins needed for any output.

### Fused outputs (require a fusion mode, not CONFIG)

| Output | Registers | Scale | Notes |
|---|---|---|---|
| Euler Heading (yaw) | 0x1A–0x1B | 1/16 deg/LSB | 0–360° range |
| Euler Roll | 0x1C–0x1D | 1/16 deg/LSB | -90 to +90° |
| **Euler Pitch** | **0x1E–0x1F** | **1/16 deg/LSB** | **-180 to +180° ← we read this** |
| Quaternion W | 0x20–0x21 | 1/2^14 per LSB | More accurate than Euler for math |
| Quaternion X | 0x22–0x23 | | |
| Quaternion Y | 0x24–0x25 | | |
| Quaternion Z | 0x26–0x27 | | |
| Linear Accel X | 0x28–0x29 | 1/100 m/s²/LSB | Acceleration minus gravity |
| Linear Accel Y | 0x2A–0x2B | | |
| Linear Accel Z | 0x2C–0x2D | | |
| Gravity X | 0x2E–0x2F | 1/100 m/s²/LSB | Gravity component only |
| Gravity Y | 0x30–0x31 | | |
| Gravity Z | 0x32–0x33 | | |

### Raw / calibrated sensor outputs (available in any mode)

| Output | Registers | Scale | Notes |
|---|---|---|---|
| **Gyro X** | **0x14–0x15** | **1/16 dps/LSB** | **Pitch rate axis ← we read this** |
| Gyro Y | 0x16–0x17 | 1/16 dps/LSB | |
| Gyro Z | 0x18–0x19 | 1/16 dps/LSB | Yaw rate |
| Accel X | 0x08–0x09 | 1/100 m/s²/LSB | Raw accel (incl. gravity) |
| Accel Y | 0x0A–0x0B | | |
| Accel Z | 0x0C–0x0D | | |
| Mag X | 0x0E–0x0F | 1/16 µT/LSB | |
| Mag Y | 0x10–0x11 | | |
| Mag Z | 0x12–0x13 | | |
| Temperature | 0x34 | 1°C/LSB | Ambient |

All values are little-endian signed 16-bit integers (LSB register, then MSB register).

## Pin Reference

### Talos wiring (current)

| BNO055 pin | Connected to | Purpose |
|---|---|---|
| GND | Nucleo GND | Ground |
| SDA | PB9 (I2C1 SDA) | I2C data |
| SCL | PB8 (I2C1 SCL) | I2C clock |
| 3Vo | PerfBoard +3V3 | 3.3V power to chip |

### All pins and what they do

| Pin | Function | Talos status | Notes |
|---|---|---|---|
| VIN | 5V power input → onboard LDO regulator | Not connected | We power directly via 3Vo, bypassing regulator. Fine for 3.3V systems. |
| 3Vo | 3.3V output from onboard regulator, OR direct 3.3V input | **Connected (power in)** | We use this as a power input, not output |
| GND | Ground | **Connected** | |
| SDA | I2C data | **Connected (PB9)** | All register reads/writes |
| SCL | I2C clock | **Connected (PB8)** | |
| RST | Hardware reset, active-low | Not connected | Pulling low resets the chip immediately. Currently we rely on the 700 ms software boot wait instead. Adding a GPIO here enables instant, reliable reset — useful if the chip locks up in field conditions. Tie to a spare GPIO in Phase 4+. |
| INT | Interrupt output, active-low | Not connected | The BNO055 can assert this pin on data-ready, any-motion, no-motion, or high-G events. Would enable interrupt-driven reading (exact timing) instead of polling. Useful in Phase 5+ when precise sample-rate sync matters. |
| ADR | I2C address select | Not connected (= LOW) | Low → address 0x28. High → 0x29. Only needed if two BNO055s share the same I2C bus. |
| PS0 | Protocol select bit 0 | Not connected (= LOW) | PS0=LOW, PS1=LOW → I2C mode. This is the default. Do not connect unless switching to UART or SPI. |
| PS1 | Protocol select bit 1 | Not connected (= LOW) | Same as above. |

**Summary: no additional wiring is needed to access any measurement output.** RST and INT are the only pins worth adding later (for robustness and precision timing respectively).

## Axis Mapping — Validated on Bench (Phase 2)

**Test conditions:** BNO055 held flat, PS0 pins at top, facing component side.

| Physical motion | Effect | Conclusion |
|---|---|---|
| PS0 edge tilts down | Euler Pitch → positive | Pitch axis confirmed |
| Opposite edge tilts down | Euler Pitch → negative | Sign confirmed |
| Left edge tilts down | Gyro X → positive spike | Gyro X = pitch rate axis ✓ |
| Left edge tilts down | Gyro Y → no response | Gyro Y is orthogonal (NOT pitch rate) |

**Axis summary:**
- BNO055 **X-axis** runs left-right of the board (perpendicular to the PS0 edge)
- Euler Pitch = rotation about the X-axis
- Gyro X = angular rate about the X-axis ← correct for pitch rate
- Gyro Y = angular rate about the Y-axis (front-back of board) ← incorrect, orthogonal

## What Talos Uses and Why (Phase 2)

```c
float pitch  = BNO055_GetPitch(&imu);   // Euler Pitch, degrees
float gyro_x = BNO055_GetGyroX(&imu);  // Gyro X, deg/s
```

These map to the balance robot state:

```
θ    = pitch    — fused tilt angle, accurate and stable long-term
θ̇   = gyro_x  — direct gyro rate, low latency, no phase lag from filter
v    = encoder-derived wheel velocity
```

**Why use both pitch and gyro_x instead of just pitch?**

The PID derivative term needs θ̇. You could compute `d(pitch)/dt` numerically, but:
- Numerical differentiation amplifies noise
- The fusion filter introduces slight phase lag in the Euler output
- Gyro X is the direct, low-latency measurement of the same physical rate

So: use fused Euler Pitch for the absolute angle reference, and raw Gyro X for the rate term.

## Mounting Guidance for Chassis Assembly

For the balance robot:
- The wheel axle runs **left-right** of the robot body
- Forward-backward tilt = rotation about the wheel axle = the pitch axis

Mount the BNO055 such that the **BNO055 X-axis is parallel to the wheel axle** (left-right of robot). Concretely: the PS0 edge of the board should point either up or down on the chassis (not toward the front or back).

**Sign convention:** with PS0 edge facing up on the mounted chassis, forward lean → Euler Pitch goes positive. Verify after first assembly. If the sign is inverted, add `PITCH_SIGN = -1` in Phase 3 code — do not modify the driver.

## Calibration

The BNO055 self-calibrates at runtime. Calibration status is in register 0x35 (CALIB_STAT), two bits per sensor:
- `0` = uncalibrated, `3` = fully calibrated

In IMU mode, only accel and gyro calibrate (no mag):
- **Gyro**: hold still for a few seconds → auto-calibrates
- **Accel**: move into several different orientations over time → auto-calibrates

Calibration is lost on power cycle in IMU mode (no NDOF persistent calibration). For balance, this is fine — gyro calibrates in seconds at startup while the robot is held still.

## Quick Register Reference

```
CHIP_ID      0x00  — should read 0xA0
OPR_MODE     0x3D  — 0x00=CONFIG, 0x08=IMU, 0x0C=NDOF
PWR_MODE     0x3E  — 0x00=NORMAL
UNIT_SEL     0x3B  — 0x00=degrees/m/s²/dps/°C (our setting)
SYS_TRIGGER  0x3F  — write 0x20 to reset chip
CALIB_STAT   0x35  — calibration status (bits 7:6=sys, 5:4=gyro, 3:2=accel, 1:0=mag)

GYR_DATA_X   0x14–0x15  ← pitch rate (we read this)
GYR_DATA_Y   0x16–0x17  ← NOT pitch rate (orthogonal)
GYR_DATA_Z   0x18–0x19  ← yaw rate

EUL_HEADING  0x1A–0x1B
EUL_ROLL     0x1C–0x1D
EUL_PITCH    0x1E–0x1F  ← pitch angle (we read this)
```

## I2C Address

Default (ADR pin low): **0x28**
STM32 HAL 8-bit format: **0x50** (= 0x28 << 1)
Alternate (ADR pin high): 0x29 / 0x52
